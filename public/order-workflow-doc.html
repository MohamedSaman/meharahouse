Order Status Reference

Every order in Meharahouse has a status that reflects exactly where it is in the fulfillment lifecycle. The table below explains each status, what it looks like, and what event caused it.

Status Code Display Label What It Means What Triggers It
new New Order just placed by the customer. No payment confirmed yet. Customer completes checkout on the website.
payment_received Payment Received Admin has confirmed the advance or full payment from the customer. Admin clicks "Confirm Receipt" after reviewing the uploaded bank slip.
confirmed Confirmed Order confirmed. All available stock has been deducted from inventory. Admin clicks "Confirm Order" and all items have sufficient stock.
sourcing Sourcing Order confirmed but one or more items are on backorder — waiting for stock to arrive. Admin clicks "Confirm Order" and at least one item was sent to "Next Batch" (backorder).
dispatched Dispatched Order has been shipped out to the customer. Shipment Batch advances to "Shipped" status, which automatically updates linked orders.
delivered Delivered Order has been received by the customer. Shipment Batch advances to "Completed". Also set when all backorders for an order are completed.
completed Completed Sale fully done. Payment marked as fully paid. No further action needed. Admin manually clicks the "Completed" button on the order detail page (only available for delivered orders).
refunded Refunded All items in the order were refunded to the customer. Every item in the order has been individually refunded, or a full manual refund was processed.
cancelled Cancelled Order has been cancelled. No further action possible. Order is cancelled by admin. If items were stocked, inventory is restored.

Payment Statuses

Each order also carries a separate payment status that tracks the financial state independently from fulfillment.

Payment Status What It Means
pending No payment received yet. Order just placed (COD or bank transfer not yet confirmed).
partial Customer paid an advance (e.g., 50%). Balance is still due on delivery.
paid Full payment received and confirmed. Also set automatically when admin marks order "Completed".
failed Payment attempt was unsuccessful or rejected.
refunded Refund has been issued back to the customer (full or partial).

Payment Flow & Statuses

Meharahouse supports two payment methods at checkout. Understanding how each flows through the system prevents confusion when processing orders.

Payment Methods

Cash on Delivery (COD)
Customer pays in cash when the order is physically delivered. No bank slip uploaded. Order starts as new with payment pending. Admin confirms order directly without needing to confirm payment first.
Bank Transfer — 50% Advance
Customer pays 50% upfront via bank transfer and uploads a payment slip. Payment status becomes partial after admin confirms. Remaining 50% is collected on delivery.
Bank Transfer — 100% Full
Customer pays the full order amount upfront. Uploads a payment slip. After admin confirms, payment status becomes paid. No balance due on delivery.

Payment Status Flow Diagram

How Admin Confirms Payment

When a customer uploads a bank slip, the order shows a "Receipt Uploaded" badge in the admin orders list. The admin should:

  1. Open the order detail page.
  2. Review the uploaded slip image.
  3. Click "Confirm Receipt" to verify the payment.
  4. The payment status updates and the order moves to Payment Received.
  5. The "Confirm Order" button then becomes available.
i
COD Orders: Cash on Delivery orders skip the payment confirmation step entirely. The "Confirm Order" button is available immediately after the order is placed, since payment is collected on delivery.

Stock Shortage Decision Guide

When admin clicks "Confirm Order" and there is not enough stock for one or more items, a Stock Alert Modal appears. For each short item, the admin must choose one of four actions.

Next Batch

The item is out of stock now but will be restocked soon. The customer's order will be fulfilled in two shipments — available items now, the rest later.

What happens: A Backorder record is created for the short quantity. The order gets status "Sourcing" until all backorders are complete.

When to use: When you know the product will arrive soon (pending purchase order or incoming shipment).

Order: SOURCING + Backorder Created
Replace

The original item is unavailable but a similar substitute product exists in stock. The customer receives a different (but equivalent) item.

What happens: You search and select a replacement product. The order item is updated inline. Price difference is calculated and order total adjusted.

When to use: When a different product meets the customer's need and you have it in stock right now.

Order: CONFIRMED + Item Replaced
Refund

The item cannot be fulfilled at all — no stock, no substitute, no restock planned. The customer is refunded for this specific item.

What happens: Item is removed from the order total. A refund is calculated proportionally based on what was paid. Refund modal may open if customer overpaid.

When to use: When the item is permanently unavailable and you cannot offer a replacement.

Order total reduced + Refund Issued
Skip (Default)

Ship whatever quantity is available now and simply do not fulfill the rest. No backorder is created. No refund is issued automatically.

What happens: Available stock is deducted and shipped. The short quantity is ignored. Order moves to CONFIRMED with reduced quantity shipped.

When to use: When partial fulfillment is acceptable and you have agreed with the customer on this.

Order: CONFIRMED (partial ship)

Stock Alert Modal: What the Admin Sees

Column What It Shows
Product NameThe item with insufficient stock
Ordered QtyHow many the customer ordered
Available QtyHow many are currently in stock
Short QtyThe deficit (ordered minus available)
Decision DropdownNext Batch / Replace / Refund / Skip selection per item
!
Important: The admin must make a decision for EVERY short item before clicking "Confirm Order" in the modal. Clicking confirm applies all decisions at once and cannot be easily undone.

Backorder Step-by-Step Guide

A backorder is created when an item is out of stock at the time of order confirmation and the admin chooses "Next Batch". Backorders are managed from the Backorders page in the admin panel. Each backorder moves through the following five stages.

1
Repurchasing

Waiting for New Stock to Arrive

The backorder is created. Admin is waiting for the supplier to deliver the item. This is the starting point for all backorders.

Admin actions at this stage:

Mark Ready — Click this when the stock arrives. System deducts the stock and moves the backorder to "Ready".

Replace — If the original product never arrives, click this to select a substitute. If the replacement is in stock, backorder becomes "Ready". If not, it stays "Repurchasing" for the new product.

Refund — If the item cannot be fulfilled at all, click this to issue a refund for the customer and cancel the backorder.

2
Ready

Stock Confirmed — Ready to Ship

The item is in stock and has been reserved for this backorder. It needs to be added to a shipment batch to go out to the customer.

Admin actions at this stage:

Go to Shipments — Navigates to the Shipments page where the admin creates or selects a batch and assigns this backorder's items to it. The batch then handles the transition to "Dispatched".

3
Dispatched

Item Sent Out

The backorder item has been included in a shipment batch and the batch has been marked as "Shipped". The item is on its way to the customer.

Admin actions at this stage:

Mark Delivered — Click this when you have confirmed the item was physically delivered to the customer. Moves to "Delivered".

4
Delivered

Customer Has Received the Item

The item has been delivered. The customer has physically received it. One final step remains to close the backorder.

Admin actions at this stage:

Completed — Click this to mark the backorder as fully done. This is also the trigger for a parent order update (see below).

5
Completed

Backorder Fully Done

The backorder is complete. No further action required.

Automatic parent order update: If this was the last outstanding backorder for the parent order, the parent order's status automatically changes to Delivered. The admin can then mark the main order as "Completed".

Cancelling a Backorder

!
Cancel Button is available at every stage. When admin clicks "Cancel" on a backorder:
  • The linked order item is marked as cancelled.
  • If the backorder was in "Ready", "Dispatched", or "Delivered" state — the reserved stock is returned back to inventory.
  • If the backorder was in "Repurchasing" — no stock was deducted, so nothing is returned.

Partial Replacement on Backorders

When replacing a backorder item, the system supports replacing only part of the quantity. For example, if a backorder is for 5 units and only 3 of the replacement product are available, the admin can replace 3 units now and create a new backorder for the remaining 2 units. The order total is recalculated with the price difference.

Refund Guide

There are two ways a refund can occur in Meharahouse. Understanding the difference helps admin handle refunds accurately and consistently.

Type A: Item Refund (Automatic during stock shortage or backorder cancellation)

This type of refund happens when a specific item cannot be fulfilled. It is triggered by selecting "Refund" in the Stock Alert Modal or by cancelling a backorder.

StepWhat Happens
1. Admin selects Refund for a short itemSystem identifies the item's unit price and the short quantity
2. Refund amount calculatedFor full-payment orders: unit_price × short_qty. For partial-payment: (item value / order total) × total paid
3. Order total recalculatedThe refunded item is removed from the order total
4. Overpayment checkIf customer paid more than the new order total, the Refund Modal auto-opens to issue the difference

Type B: Manual Order Refund

Admin can manually issue a refund for any reason at any point from the order detail page.

1

Open Order Detail → Click "Refund" Button

Available on any order that has received payment. Opens the Refund Modal.

2

Fill in Refund Details

Enter: Refund Amount, Method (Bank Transfer / Cash / Online), Bank Account details, and Reference Number.

3

Refund Record Created (Status: Pending)

A refund record is logged in the system with "Pending" status. The actual bank transfer has not been made yet.

4

Go to Refunds Page → Complete the Transfer

Admin goes to the Refunds management page, finds the pending refund, and marks it as completed once the money has been sent to the customer.

Refund Calculation Rules

ScenarioMaximum Refundable Amount
Full payment orderTotal amount paid minus any refunds already issued
Partial payment (50% advance)Proportional: (item value / order total) × amount paid
COD order (cash collected)Full order total (admin collected cash offline)
!
Partial Payment Note: For orders where the customer only paid a 50% advance, the system will only refund the proportional share of that advance. You cannot refund more than what the customer actually paid. The unpaid balance is simply cancelled and not charged.

Replace Guide

The Replace feature allows admin to substitute an unavailable product with an alternative. It can be used in two different contexts, each with slightly different behavior.

Context A: Inline Replace (During Stock Shortage Modal)

Used when confirming an order that has stock shortages. The replacement happens immediately as part of the order confirmation process.

StepAction
1Admin selects "Replace" for the short item in the Stock Alert Modal
2A product search field appears — admin searches and selects a substitute product
3System verifies the replacement product has sufficient stock
4Order item is updated immediately with the new product
5Price difference is calculated and the order total is adjusted

Price Difference Handling (Both Contexts)

ScenarioWhat Happens to Order TotalAction Required
Replacement is cheaper Order total is reduced by the price difference If customer overpaid, Refund Modal opens automatically to refund the difference
Replacement is more expensive Order total increases by the price difference Customer owes a balance — admin should communicate this to the customer
Replacement is same price Order total unchanged No financial adjustment needed

Context B: Backorder Replace (From Backorders Page)

Used when the original product in a backorder never arrives and admin needs to source an alternative.

StepAction
1Admin opens the Backorders page and finds the backorder in "Repurchasing" status
2Admin clicks the "Replace" button on the backorder
3Admin searches and selects a substitute product
4If replacement IS in stock: backorder moves to "Ready" — can now be dispatched
5If replacement is NOT in stock: backorder stays "Repurchasing" (now tracking the new product)
6For partial replacement: admin specifies quantity to replace, system creates a new backorder for remainder
7Order total is recalculated with the price difference from the substitution
v
Best Practice: Always communicate with the customer before replacing an item. Inform them of the substitute product and any price changes before finalizing the replacement in the system.

Quick Reference Card

A concise summary of every button in the system, where to find it, and what it does. Use this as a daily reference guide.

Orders List Page

Confirm Receipt → Verifies bank slip. Order → Payment Received.
Confirm Order → Checks stock and confirms. Order → Confirmed or Sourcing.
Completed → Marks delivered order as done. Payment → Paid.
Refund → Opens manual refund modal for any paid order.

Stock Alert Modal (During Confirm Order)

Next Batch → Creates backorder for short items. Order → Sourcing.
Replace → Swaps item with in-stock substitute inline.
Refund → Removes item from order and refunds customer.
Skip → Ships available qty only, no backorder, no refund.

Backorders Page

Mark Ready → Stock arrived. Deducts stock. BO → Ready.
Replace → Swap original product. BO → Ready (if in stock).
Refund → Cancel BO, refund customer for this item.
Go to Shipments → Navigate to Shipments page to assign BO to batch.
Mark Delivered → Customer received BO item. BO → Delivered.
Completed → Close BO. If last BO, parent order → Delivered.
Cancel → Cancel BO. Returns stock if was deducted.

Shipments Page

Create Batch → New shipment batch with name, courier, tracking.
Assign Orders → Add confirmed orders to the batch.
Mark Shipped → Batch shipped. All linked orders → Dispatched.
Mark Completed → Batch done. All linked orders → Delivered.

Refunds Page

Complete Refund → Confirms money was sent. Refund record → Completed.

Order Status Progression at a Glance

New → Customer places order
Payment Received → Admin confirms slip
Confirmed → Stock deducted, ready to ship
Sourcing → Has active backorders
Dispatched → Batch shipped
Delivered → Batch completed / all BOs done
Completed → Admin finalizes sale
i
Key Principle: Stock is never deducted when an order is placed. It is only deducted when admin clicks "Confirm Order" (for available items) or "Mark Ready" (for backorder items). This prevents stock from being locked by unverified orders.
v
Normal Workflow Summary: Customer places order → Admin confirms payment → Admin confirms order (stock check) → Assign to Shipment Batch → Batch ships → Batch completes → Admin marks Completed. Each step is deliberate and admin-controlled to ensure accuracy.
